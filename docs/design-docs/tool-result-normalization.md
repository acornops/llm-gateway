# Tool Result Normalization

The gateway is the trust boundary between MCP transports and the execution engine. Its normalized response separates `full_result`, `model_context`, `context_meta`, `artifact_eligible`, and `is_error`.

Trusted AgentK tools are identified by built-in registry provenance and Kubernetes target type. They must have an advertised output schema and return matching `acornops.model-context.v1` and `acornops.full-tool-result.v1` envelopes. The gateway validates the schema version, expected tool identity, required context fields, error-status consistency, 500-character summary, exact transmitted context size, declared size bounds, independently computed 12 KiB context and 2 MiB full-result limits, metadata shape, output schema, and registry-owned artifact policy. The full-result declaration is not compared byte-for-byte across JavaScript and Python serializers; the parsed result is measured independently so valid exponent formatting cannot create a false contract failure. Any mismatch becomes `TOOL_RESULT_SCHEMA_INVALID`; trusted results never fall back to generic compaction.

Other built-in tools, including AgentV, and third-party MCP tools use `content` for model evidence and `structuredContent` as the complete view when present. Their artifact policy is always untrusted, so they cannot enable retention. The execution engine applies deterministic structural fallback to these results. Missing or malformed MCP result fields become `TOOL_RESULT_CONTRACT_INVALID` instead of an empty successful observation.

Complete result data is subject to the 2 MiB uncompressed ceiling. Remote MCP responses use that same 2 MiB transport ceiling because they do not carry retained trusted envelopes. Built-in AgentK responses have a separate 3 MiB transport-envelope ceiling, leaving bounded room for the 2 MiB result plus model context and protocol metadata. Both HTTP paths are decoded through bounded streams and closed as soon as their ceiling is crossed, so an oversized peer cannot force an unbounded response allocation. Registry metadata—not MCP response metadata—is authoritative for artifact eligibility.

Built-in transport deadlines include five seconds of headroom beyond the producer-advertised tool deadline. This lets AgentK return its structured timeout code, retryability, operation ID, and unknown-outcome marker instead of racing the gateway into a generic transport error. The headroom does not extend AgentK's execution deadline.

If a transport fails or returns a malformed result after dispatch, the gateway marks every write-capable tool result with `outcome: unknown`. Failures proven to occur before dispatch are marked `not_started` and may be retried according to their retryability. In particular, the control-plane built-in bridge returns a retryable `TARGET_AGENT_UNAVAILABLE` with `outcome: not_started` when no target-agent connection can receive the command, and the gateway preserves that structured result instead of collapsing it into a generic 502. The execution engine must stop on unknown outcomes and require target inspection before any retry; a network failure cannot prove that a mutation did not occur.

The normalized gateway response contract is capped by the execution engine at 5 MiB. This bounded headroom is intentional: for a generic MCP server, the same result may occupy both `full_result` and the pre-fallback `model_context` until the engine applies structural compaction.
